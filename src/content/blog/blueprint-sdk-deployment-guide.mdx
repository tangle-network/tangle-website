Blueprint SDK deployment turns a local Rust service into a Tangle service that operators can register for and run. The important path is not only "write Rust, deploy contract." A production-ready Blueprint needs a service definition, a runtime, a deployable artifact, operator settings, tests, and a rollout plan for testnet before mainnet.

This guide focuses on the practical deployment path. ## What The Blueprint SDK Deploys

A Blueprint is a reusable service definition. Developers define jobs, metadata, sources, and runtime behavior. Operators run the Blueprint. Consumers create service instances and send work.

| Layer | What it does |
| --- | --- |
| Blueprint definition | Describes metadata, jobs, artifact sources, and manager settings |
| Rust service | Implements the job handlers and runtime logic |
| Operator runtime | Runs the service with keys, RPC URLs, and environment settings |
| Tangle registration | Publishes the Blueprint so operators and users can discover it |
| Service instance | The customer-facing instance that receives work |

The SDK source lives at [github.com/tangle-network/blueprint](https://github.com/tangle-network/blueprint). The CLI is `cargo-tangle`, and the Rust crates are published under the Blueprint SDK workspace.

## Local Deployment Path

Start with the local loop. A broken local service becomes a harder broken testnet service.

```bash
cargo install cargo-tangle --git https://github.com/tangle-network/blueprint --force
cargo tangle blueprint create --name my_blueprint
cd my_blueprint
cargo build
cargo test
```

The create command scaffolds a Blueprint project. Build proves the Rust code compiles. Tests should exercise the job handlers and any service manager logic before the Blueprint is registered anywhere.

### Define Jobs And Sources

The definition file is the contract between the developer, operators, and deployment tooling. At minimum it needs metadata, at least one job, at least one source, and enough integrity information for the artifact. The Blueprint SDK supports machine-readable definition files so deployment tooling can validate what is being published.

Do not treat the definition as documentation only. It is deployment input.

### Run With Local Settings

A local operator run needs settings such as RPC URLs, keys, service ports, and any job-specific environment variables. Keep those values out of source control and test them before testnet registration.

The safe local loop is:

1. Build the service.
2. Run unit and integration tests.
3. Start the runner with local settings.
4. Submit one known job.
5. Confirm logs, output, and error behavior.

## Testnet Deployment

Once the local loop is clean, deploy to Tangle testnet:

```bash
cargo tangle blueprint deploy tangle \
  --network testnet \
  --definition ./definition.json
```

Use testnet to prove four things:

| Testnet check | What success looks like |
| --- | --- |
| Registration | Blueprint metadata is discoverable |
| Operator run | At least one operator can start the service |
| Job execution | A sample job succeeds end to end |
| Error path | Bad input fails with a useful error |
| Observability | Logs and metrics identify the service instance |

Testnet is where you catch packaging mistakes, missing environment variables, bad source URLs, wrong ports, and operator setup drift.

## Production Deployment Checklist

Before mainnet, the Blueprint should have a documented operating envelope.

| Area | Production requirement |
| --- | --- |
| Artifact | Immutable source, image, or binary with integrity metadata |
| Keys | Operator keys separated from developer keys |
| RPC | Stable endpoints and fallback behavior |
| Pricing | Clear job pricing or quote path |
| Verification | The right mechanism for the job type |
| Monitoring | Logs, metrics, health checks, and incident response |
| Rollback | Known downgrade or redeploy path |

For paid services, connect this with [x402 payments for AI agents](/blog/x402-payments-for-ai-agents) and [Blueprint SDK x402 payments](/blog/blueprint-sdk-x402-payments-runnable-jobs). For verification, read [how Tangle verifies work](/blog/how-tangle-verifies-work).

### Mainnet Should Follow Evidence

Mainnet deployment should be a promotion from evidence, not a ceremony. The service should already have successful testnet jobs, operator runbooks, artifact integrity checks, and a rollback plan.

That is especially true for services with payments, private data, trusted execution environments, or slashing exposure. Those systems need separate gates because failure can cost money or leak data.

### What To Automate

Automate the checks that prevent bad deploys:

- Build and test before deployment.
- Validate the definition file.
- Confirm artifact availability.
- Confirm operator settings schema.
- Submit a smoke job after registration.
- Record the deployed Blueprint ID, network, artifact hash, and git commit.

The goal is not to make deployment mysterious. The goal is to make each promotion repeatable.

## FAQ

### What is Blueprint SDK deployment?

Blueprint SDK deployment is the process of registering a Blueprint service definition, publishing its artifact, and running it through Tangle operators.

### What does cargo-tangle do?

`cargo-tangle` is the CLI used to create Blueprint projects and deploy Blueprint definitions to Tangle networks.

### Should I deploy to testnet first?

Yes. Testnet proves registration, operator startup, job execution, logs, and failure behavior before money or production users depend on the service.

### What belongs in a Blueprint definition file?

The definition should describe metadata, jobs, artifact sources, manager settings, and integrity information for the deployed service.

### How is this different from deploying a normal API?

A normal API is usually run by one operator. A Blueprint is a service definition that multiple operators can register for and run under Tangle's protocol rules.
