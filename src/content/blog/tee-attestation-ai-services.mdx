TEE attestation for AI services proves properties of the execution boundary: what code was loaded, what hardware-backed environment is in use, and whether a verifier can bind a result to that environment. It does not prove the model chose the right action, the answer is correct, or the service deserves payment. In Tangle, attestation is one control in a larger trust model with Blueprint execution, x402 payment, operator health, traces, and result verification.

Precision matters here. "Trusted execution" is useful only when everyone knows what is being trusted.

## Proves / Does Not Prove

| Claim | TEE attestation can prove? |
| --- | --- |
| code identity | yes, if measurement and verification are wired correctly |
| hardware-backed execution boundary | yes, subject to platform guarantees |
| secret handling boundary | partly, with correct key release policy |
| model reasoning quality | no |
| task result correctness | no |
| payment fairness | no, needs payment and result verification |

## Where It Fits

TEE attestation sits between deployment and verification:

```bash
curl -fsS https://agents.tangle.tools/.well-known/tangle-agent.json
curl -fsS https://router.tangle.tools/.well-known/tangle-agent.json
```

Those discovery calls do not attest code. They show where an agent can find service contracts. Actual attestation must bind a service identity, measured code, hardware report, verifier policy, and result path.

## Attestation Review Checklist

Before accepting a TEE claim, ask for the binding points:

| Question | Why it matters |
| --- | --- |
| what code measurement is expected? | prevents "some enclave ran something" claims |
| who verifies the report? | identifies the trust root |
| when are secrets released? | separates boot trust from runtime policy |
| how is the result tied to the run? | avoids detached attestation artifacts |
| what checks task quality? | keeps runtime trust separate from correctness |

That last question is the one buyers usually skip. TEE narrows where code ran; it does not grade what the code produced.

## Pair It With Result Verification

The production trust model should separate two proofs:

| Proof | Question answered |
| --- | --- |
| attestation | did expected code run inside the expected boundary? |
| result verification | did the service produce an acceptable result? |

For deterministic jobs, result verification may be a replay, checksum, or on-chain check. For AI jobs, it may be an eval, human review, policy check, or downstream acceptance test. Conflating those two proofs is how "trusted execution" becomes marketing instead of engineering.

## External Reference Points

The underlying category is documented by hardware and cloud providers: [Intel SGX](https://www.intel.com/content/www/us/en/products/docs/accelerator-engines/software-guard-extensions.html), [AMD SEV-SNP](https://www.amd.com/en/developer/sev.html), and Google Cloud [Confidential Computing](https://cloud.google.com/confidential-computing/docs). Those sources describe execution boundaries, not agent task quality.

## What This Does Not Prove

Attestation does not prove the output is correct. A malicious or poorly specified task can run inside a trusted boundary. A model can hallucinate inside a trusted boundary. A paid job can execute in a TEE and still need result checks before settlement or reputation changes.

## Start

Use attestation as one control, not the whole trust model. Pair it with result verification, payment verification, operator health, and trace review before treating an AI service as production-grade.

## FAQ

### What does TEE attestation prove for AI services?

It can prove code identity, hardware-backed execution state, and the execution boundary used by the service.

### Does TEE attestation prove an AI result is correct?

No. It proves execution properties, not task quality or reasoning correctness.

### How should Tangle builders use attestation?

Use it with Blueprint execution, x402 payment checks, operator health, traces, and result verification.
